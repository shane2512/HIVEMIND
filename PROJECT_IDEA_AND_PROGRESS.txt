Date: 2026-03-19
Status: Live multi-agent pipeline running on Hedera Testnet with local LLM-first reasoning and successful end-to-end validation.

====================================================================
1) Vision and Product Goal
====================================================================

HIVE MIND is a Hedera-native autonomous intelligence system designed to evaluate token activity and publish structured risk intelligence through HCS.

Primary goal:
- Turn token-event detection into a repeatable, auditable, and mostly autonomous due-diligence workflow.

Why this matters:
- Most token-review workflows are manual, fragmented, and difficult to audit.
- HIVE MIND makes the full decision chain observable through on-ledger message flow.
- The architecture supports progressive intelligence upgrades (LLM-first reasoning) while preserving deterministic fallback for reliability.

Core outcomes expected from the product:
- Event ingestion from Hedera ecosystem signals.
- Pipeline planning based on registered capabilities and budget constraints.
- Parallelized analytical stage execution.
- Unified risk scoring and publication of a final HIVE_REPORT.
- A verifiable, topic-driven execution trail from trigger to settlement.

====================================================================
2) System Architecture Overview
====================================================================

Main agent roles:
- watcher:
  - Detects qualifying tokens (age/liquidity filters plus LLM triage).
  - Publishes TASK_BUNDLE.

- plumber:
  - Loads agent manifests from registry topic.
  - Builds and publishes PIPELINE_BLUEPRINT or PIPELINE_FAILED.
  - Tracks pipeline completion counts and optional NFT mint thresholds.

- wallet-analyst:
  - Performs wallet/account behavior analysis.
  - Publishes TASK_ATTESTATION with WalletHistory output summary.

- sentiment:
  - Scans topic/message context for token mention sentiment.
  - Publishes TASK_ATTESTATION with SentimentScore output summary.

- liquidity:
  - Computes concentration and liquidity indicators.
  - Publishes TASK_ATTESTATION with LiquidityReport output summary.

- risk-scorer:
  - Aggregates upstream outputs into normalized risk.
  - Publishes TASK_ATTESTATION with RiskScore output summary.

- report-publisher:
  - Builds final HIVE_REPORT.
  - Publishes TASK_ATTESTATION and PIPELINE_COMPLETE.

Execution order:
- Stage 0 in parallel: wallet-analyst, sentiment, liquidity
- Stage 1: risk-scorer
- Stage 2: report-publisher

Topic-oriented protocol model:
- TASK_BUNDLE -> PIPELINE_BLUEPRINT -> TASK_ATTESTATION chain -> HIVE_REPORT -> PIPELINE_COMPLETE

3) LLM Strategy and Reasoning Model
====================================================================

Design principle:
- LLM-first reasoning with deterministic fallback.

What LLM-first means in this project:
- The default path asks local Ollama model phi3.5 for structured reasoning and/or summary.
- If LLM returns invalid output, times out, or is disabled, deterministic logic keeps pipeline continuity.

Shared LLM utility:
- askLocalText and askLocalJson wrappers for local inference calls.
- JSON extraction and validation helpers.
- Confidence normalization and output sanitation patterns.

LLM-first coverage by component:
- watcher triage decision.
- plumber stage selection and planning rationale.
- wallet behavior interpretation.
- sentiment classification.
- liquidity risk interpretation.
- risk-score synthesis.
- report summary composition.

Reliability controls:
- Supports confidence values in either 0-1 or 0-100 range.
- Validates expected JSON schema before accepting model output.
- Applies guardrails when model output could contradict basic risk sanity.

5) Runtime Hardening and Bug Fixes Applied
====================================================================

1. Confidence normalization
- Problem: mixed confidence scales from model outputs.
- Fix: normalized to bounded 0-100 consistently.

2. Signal serialization
- Problem: object-shaped signals rendered as non-useful values.
- Fix: normalized to readable string signals.

3. Liquidity guardrails
- Problem: edge outputs could understate concentration risk.
- Fix: deterministic safety limits for extreme concentration scenarios.

4. Plumber resilience
- Problem: malformed TASK_BUNDLE could stall processing loop.
- Fix: malformed bundles are skipped and converted into PIPELINE_FAILED path instead of blocking run.

5. Worker lookup/read depth
- Problem: under topic churn, task lookups could miss related bundles.
- Fix: larger read window and configurable lookup depth.

6. Verify resilience
- Problem: transient fetch failures could abort verification early.
- Fix: retry within polling loop and continue until timeout.

7. Runbook determinism improvements
- Seeded TASK_BUNDLE after token creation.
- Optional watcher inclusion mode.
- State reset behavior for clean runs by default.

9) Known Constraints and Tradeoffs
====================================================================

1. High event volume can introduce queueing delay
- Mitigated by deterministic runbook mode and state reset options.

2. LLM output variability
- Mitigated by validators, normalization, and deterministic fallback.

3. External network/mirror intermittency
- Mitigated by retry-aware verification and loop reconnection behavior.

4. Planner quality depends on manifest quality
- Better manifest metadata yields more reliable stage plans.
