HIVE MIND: Detailed Idea and Progress Report
Date: 2026-03-22
Status: Live multi-agent Hedera pipeline with observer UI, lifecycle-based external agent onboarding, and operational one-command runtime launchers.

====================================================================
1) Vision and Product Goal
====================================================================

HIVE MIND is a Hedera-native autonomous intelligence pipeline that detects token activity, coordinates specialized agents, and publishes auditable risk intelligence to HCS.

Primary goal:
- Convert token-event detection into a repeatable, inspectable, and mostly autonomous due-diligence workflow.

Why this matters:
- Manual token review is fragmented and hard to audit.
- HIVE MIND preserves the full decision chain on Hedera topics.
- The architecture supports LLM-first intelligence with deterministic fallback for reliability.

Core outcomes:
- Event ingestion from Hedera ecosystem signals.
- Planner-driven pipeline assembly from registry capabilities.
- Parallel analytical execution.
- Unified scoring and final HIVE_REPORT publication.
- Verifiable topic trail from trigger to completion.

====================================================================
2) Current Architecture Overview
====================================================================

Main agent roles:
- watcher:
  - Detects candidate tokens (filters plus optional LLM triage).
  - Publishes TASK_BUNDLE.

- plumber:
  - Loads AGENT_MANIFEST entries from registry topic.
  - Builds and publishes PIPELINE_BLUEPRINT or PIPELINE_FAILED.
  - Tracks completion behavior and downstream orchestration state.

- wallet-analyst:
  - Performs wallet/account behavior analysis.
  - Publishes TASK_ATTESTATION with wallet output summary.

- sentiment:
  - Classifies topic/message context and sentiment posture.
  - Publishes TASK_ATTESTATION with SentimentScore summary.

- liquidity:
  - Computes concentration and liquidity indicators.
  - Publishes TASK_ATTESTATION with LiquidityReport summary.

- risk-scorer:
  - Aggregates upstream outputs into normalized risk.
  - Publishes TASK_ATTESTATION with RiskScore summary.

- report-publisher:
  - Assembles final HIVE_REPORT.
  - Publishes TASK_ATTESTATION and PIPELINE_COMPLETE.

Nominal execution order:
- Stage 0 in parallel: wallet-analyst, sentiment, liquidity
- Stage 1: risk-scorer
- Stage 2: report-publisher

Topic-oriented protocol chain:
- TASK_BUNDLE -> PIPELINE_BLUEPRINT -> TASK_ATTESTATION chain -> HIVE_REPORT -> PIPELINE_COMPLETE

3) Technology Stack and Why It Was Chosen
====================================================================

Runtime and language:
- Node.js >= 18 (JavaScript, CommonJS)
- Why: fast iteration for multi-process agent workers, rich SDK ecosystem, straightforward local scripting.

Ledger and network services:
- Hedera Consensus Service (HCS)
  - Why: immutable, ordered, auditable message stream for registry, tasks, attestations, and reports.
- Hedera Token Service (HTS)
  - Why: native token operations for PIPE economics and test token lifecycle.
- Hedera EVM + contracts (Hardhat)
  - Why: escrow/payment trust enforcement where programmable on-chain logic is needed.

Core libraries:
- @hashgraph/sdk
  - Why: native Hedera integration for topic writes, reads, and token operations.
- hedera-agent-kit
  - Why: agent-oriented Hedera utility integration.
- langchain, @langchain/core, @langchain/langgraph, @langchain/ollama
  - Why: structured LLM orchestration and local model routing.
- ollama
  - Why: local phi3.5 inference path for private, low-latency reasoning.
- dotenv
  - Why: environment-driven runtime config across scripts/agents/UI.
- ethers
  - Why: EVM interaction support in contract/payment flows.

Frontend and visualization:
- React + Vite observer UI
  - Why: rapid UI iteration, lightweight build pipeline, responsive dashboard rendering.

Packaging and distribution:
- npm package: hivemindregistration (v0.1.4)
  - Why: simple external-agent onboarding UX through one CLI install and one command flow.

4) LLM Strategy and Reliability Model
====================================================================

Design principle:
- LLM-first reasoning with deterministic fallback.

Meaning in practice:
- Default path uses local Ollama model (phi3.5) for structured reasoning/summaries.
- Invalid/timeout/offline model output falls back to deterministic logic.

Reliability controls:
- Confidence normalization and schema checking.
- Guardrails for high-risk concentration edge cases.
- Structured output validation before acceptance.

Coverage:
- watcher triage
- plumber planning rationale
- worker interpretation and synthesis
- risk scoring and report narrative augmentation

5) Protocol and Registry Model (Updated)
====================================================================

Supported lifecycle message types now include:
- AGENT_REGISTER
- AGENT_CLAIMED
- AGENT_HEARTBEAT
- AGENT_REVOKED
- AGENT_MANIFEST

These are validated in shared UCP schema handling.

Current operational behavior:
- Scheduler/planner discovery is still manifest-centric.
- Lifecycle events provide stronger operational governance (claim/lease/revoke) and are used by onboarding and cleanup workflows.

Practical "remove agent" semantics:
- Topic history is immutable; deletion is not possible.
- Effective removal is AGENT_REVOKED as latest state for an agent.

7) Runtime Orchestration and Startup UX
====================================================================

Added startup modes in root scripts:
- all:start
  - Starts 7 agents plus UI from one command in a shared process runner.
- all:start:separate
  - Starts each agent and UI in separate PowerShell windows.
- all:start:separate:dry
  - Prints launch commands without starting processes.

Why both modes exist:
- all:start: easier one-process control and shutdown.
- all:start:separate: better per-agent visibility and debugging isolation.

10) Known Constraints and Tradeoffs
====================================================================

1. Mirror indexing delay
- New topic events can appear with short lag.
- Mitigation: retry-aware verify loop.

2. LLM output variability
- Mitigation: schema guards, normalization, deterministic fallback.

3. High topic churn/read windows
- Mitigation: tuned read depth and robust loop behavior.

4. Immutable history model
- Agents cannot be deleted from history.
- Operational deactivation relies on AGENT_REVOKED latest state.
